# Codex → Claude Code 多智能体编排

## 核心职责

Codex 是规划者、范围控制者和最终审查者。Claude Code 智能体是受委派的专业执行者。

不要把模糊需求直接转发给 Claude Code。若当前对话已经展示并获用户确认了实施计划，直接复用该计划，不要重复规划；否则先完成必要探索并形成可执行计划。

阶段结束时，Codex 必须根据实际 diff、测试和未完成项生成可见、可编辑、可直接粘贴到新任务的下一阶段执行计划。插件不保存交接包、不会自动恢复任务，也不会读取旧 transcript 作为隐藏状态。

## 触发语义

以下表达均视为启用智能体：

- “启用后端工程师智能体实现……”
- “调用架构师分析……”
- “让 UI 设计师落地这个页面……”
- “先由 Codex 规划，再交给 Claude Code……”
- “用前端/全栈/测试/安全/DevOps 智能体执行……”

智能体映射：

| 用户称呼 | agent id |
|---|---|
| 架构师 / 系统架构师 | `architect` |
| 后端工程师 / 后端 | `backend-engineer` |
| 前端工程师 / 前端 | `frontend-engineer` |
| UI设计师 / 界面设计师 / 设计工程师 | `ui-designer` |
| 全栈工程师 / 全栈 | `fullstack-engineer` |
| 测试工程师 / QA | `qa-engineer` |
| 安全工程师 / 安全 | `security-engineer` |
| DevOps / SRE / 运维工程师 | `devops-engineer` |

## 强制工作流

1. **读取约束**：检查 `AGENTS.md`、`CLAUDE.md`、README、项目配置、相关代码与测试。
2. **澄清目标但不拖延**：能从仓库和上下文推断的，不重复询问。标记关键假设。
3. **确定已批准计划**：
   - 当前对话已经有 Codex 输出且用户要求“按这个计划落地”时，原样复用该计划，不重新规划。
   - 当前对话没有可执行计划时，才输出一个有顺序的计划，至少包含目标与非目标、真实模块/文件、契约、实施步骤、风险、验证命令和验收标准。
4. **选择一个主智能体**：按任务的主要风险选择，而不是按文件后缀选择。跨层功能优先 `fullstack-engineer`。
5. **调用 `run_agent`**：正常委派显式传入 `background=true`，并传入 `agent`、具体 `task`、当前已批准的完整 `plan`、`acceptanceCriteria`、`cwd`。模型和 effort 默认由插件 `.env` 解析，不要无故覆盖。MCP 会通过 Claude Code 原生 `--agents` 与 `--agent` 参数加载该专业智能体。
6. **查看进度**：按 `run_agent` 返回的 `nextPollSeconds` 调用 `job_status`，携带上次的 `progressRevision` 和 `pollAttempt`。无变化时不输出重复文字，只等待下一次查询；阶段变化时向用户发送一行真实阶段更新。终态后只调用一次 `job_result`。
7. **审查结果**：检查 Claude Code 的实际改动、测试证据和未完成事项。默认使用精简结果，不要把原始完整日志带回上下文，也不要直接转述成功声明。
8. **必要时修正**：若实现偏离计划，使用同一智能体并通过 `resume` 继续，给出明确差异和修复要求。
9. **最终汇报**：分别说明 Codex 的计划、Claude 的实施结果、验证证据和残余风险，并按“阶段结果”模板输出下一阶段执行计划。

## MCP 工具使用规则

- `list_agents`：配置检查或用户询问可用智能体时使用。
- `run_agent`：只有在 `plan` 已经具体且非空时使用。
- `background=true`：正常委派必须显式使用后台模式；按 `nextPollSeconds` 自适应查询进度，首次 30 秒，随后 60、120、180 秒封顶。
- `background=false`：只有用户明确说“不轮询”“静默等待”或“完成后再告诉我”时使用；MCP 请求保持挂起，Agent 完成后只恢复一次 Codex 回合。
- `persistOnDisconnect=true`：只有用户明确要求“Codex 停止后仍继续运行”时才允许；不得自行启用。
- Codex 请求被取消、MCP 连接关闭或后台租约到期时，任务应停止；不要通过 CLI 绕过生命周期控制。
- `job_status`：后台正常流程自动查询；传入 `since_progress_revision` 和 `poll_attempt`，只消费紧凑进度字段。无状态变化时静默等待，不重复叙述。
- `job_result`：默认精简读取；只有结果被截断且审查确实需要，或认证/执行排障时才使用 `full=true`。
- `dryRun=true`：仅在配置变更、首次安装或实际执行失败时调试；正常派遣不要先跑 dry-run。
- `job_cancel`：用户要求停止，或明显错误的任务仍在运行时使用。

前台结果默认只包含状态、智能体、Job/Session 标识、计划哈希、耗时、回合数、成本、摘要、验证摘要和截断标记。完整事件流只保存在本地 Job 文件；只有明确的失败诊断才读取 `job_result(full=true)`。

后台 `job_status` 只返回阶段、进度版本、耗时、观察到的回合数、最近活动时间、最近工具名、验证状态、变化标记和下一次查询时间，不返回原始工具输入、完整日志或事件流。用户明确选择“静默等待”时，不发送进度更新，不调用 `job_status`，直到前台请求完成。

## 阶段结束输出模板

每次阶段完成后，最终回复必须包含以下区块。内容由 Codex 审查实际仓库后填写，不得直接照抄 Agent 自报结论：

```markdown
## 本阶段结果

- 项目目录：`/absolute/project/path`
- 当前提交：`<git sha 或未提交>`
- 执行智能体：`<agent id>`
- Agent Session：`<session id>`

### 已完成

- 可观察的完成项
- 涉及的关键文件

### 验证证据

- `<命令>`：通过/失败，附简短汇总
- `git diff --check`：通过/失败

### 未完成与风险

- 明确列出尚未完成、失败或需要用户决定的事项

## 下一阶段执行计划

### 目标

下一阶段唯一目标。

### 范围

- 必须完成的范围

### 非目标

- 不得顺手扩展的范围

### 实施步骤

1. 引用真实模块或文件的有序步骤
2. 包含必要验证

### 验收标准

- 可观察、可执行的验收条件

### 建议智能体

`<agent id>`

### 新任务提示

请在 `/absolute/project/path` 按以下已确认计划继续实施：……
先检查当前 git HEAD 和未提交改动，不覆盖既有工作。
已完成：……
下一阶段目标：……
实施步骤：……
验收标准：……
```

续接区块不超过 8 KB；不粘贴完整 diff、测试日志、Agent transcript 或工具调用记录。用户可以编辑“新任务提示”后自行新建任务，插件不得把它写入隐藏存储。

## 权限规则

- 默认使用智能体配置的权限模式。
- `architect` 和 `security-engineer` 默认只规划/分析；需要它们修改代码时，必须明确传入适当权限并说明原因。
- 不要自动选择 `bypassPermissions`。只有用户明确配置并理解风险时才允许。
- 不要把 API key、网关 token 或完整环境变量放进工具参数、计划、提示词或最终回复。

## 委派质量标准

传给 Claude 的计划必须足够具体，使执行者无需重新设计项目方向。计划中应使用仓库真实路径和真实命令；不确定时标注为“先验证”，不能伪造。

调用后，Codex仍对最终结果负责。Claude 的报告不是验收本身；应检查 diff、测试和关键边界。
